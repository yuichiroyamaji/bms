# CI/CD Setup Guide

> 🤖 This documentation was generated by Kiro AI

Complete guide to set up automated deployments with GitHub Actions.

## Overview

The CI/CD pipeline automatically deploys your application when you push code:
- **`develop` branch** → Development environment
- **`main` branch** → Production environment (with approval)

## Prerequisites

- GitHub repository
- AWS account with CDK bootstrapped
- Docker installed locally (for testing)

## Setup Steps

### 1. Deploy Infrastructure Stack

First, deploy the InfraStack to create the GitHub OIDC role:

```bash
cd infra
npm run build

# Deploy InfraStack (creates GitHub OIDC role)
npx cdk deploy InfraStack-dev -c environment=dev
```

This will output the IAM Role ARN:
```
Outputs:
InfraStack-dev.GitHubOidcGitHubActionsRoleArn = arn:aws:iam::ACCOUNT:role/...
```

### 2. Add GitHub Repository Secrets

Go to your GitHub repository:
**Settings → Secrets and variables → Actions → New repository secret**

Add these two secrets:

**Secret 1: AWS_ROLE_ARN**
```
arn:aws:iam::ACCOUNT:role/admin-infra-dev-GitHubOidcGitHubActionsRole...
```
(Copy from CDK output above)

**Secret 2: AWS_REGION**
```
ap-northeast-1
```
(Or your preferred AWS region)

### 3. Set Up GitHub Environments (Optional but Recommended)

For production approval workflow:

1. Go to **Settings → Environments**
2. Create environment: **`production`**
3. Enable **"Required reviewers"**
4. Add yourself or team members as reviewers

This ensures production deployments require manual approval! 🛡️

### 4. Deploy Development Environment

```bash
cd infra
npm run deploy:dev
```

This creates your dev AppRunner service.

### 5. Deploy Production Environment (When Ready)

```bash
cd infra
npm run deploy:prod
```

This creates your prod AppRunner service.

## How It Works

### Automatic Deployments

**Push to `develop` branch:**
```bash
git checkout develop
git add .
git commit -m "Update feature"
git push origin develop
```
→ Automatically deploys to dev environment

**Push to `main` branch:**
```bash
git checkout main
git merge develop
git push origin main
```
→ Triggers production deployment (requires approval if configured)

### Manual Deployments

1. Go to **Actions** tab in GitHub
2. Select **"Deploy to AWS AppRunner"**
3. Click **"Run workflow"**
4. Choose environment (dev or prod)
5. Click **"Run workflow"**

## Workflow Steps

The GitHub Actions workflow performs:

1. ✅ Checkout code
2. ✅ Setup Node.js 20
3. ✅ Configure AWS credentials (via OIDC)
4. ✅ Install dependencies
5. ✅ Build Next.js app
6. ✅ Build infrastructure code
7. ✅ Preview changes (CDK diff)
8. ✅ Deploy to AppRunner

## Troubleshooting

### Deployment Fails with "Could not assume role"

**Issue**: GitHub can't assume the AWS IAM role

**Solution**:
1. Verify `AWS_ROLE_ARN` secret is correct
2. Check the role trust policy includes your GitHub repo
3. Ensure the role has necessary permissions

### Deployment Fails with "Stack does not exist"

**Issue**: Target stack hasn't been created yet

**Solution**:
```bash
# Create the stack manually first
cd infra
npm run deploy:dev  # or deploy:prod
```

### Docker Build Fails

**Issue**: Out of memory during build

**Solution**:
- GitHub Actions runners have 7GB RAM
- If build fails, optimize your Next.js build
- Consider using smaller dependencies

### Production Deployment Stuck

**Issue**: Waiting for approval

**Solution**:
1. Go to Actions → Click on the workflow run
2. Click **"Review deployments"**
3. Select **production** environment
4. Click **"Approve and deploy"**

## Security Best Practices

### ✅ What We Do

- Use OIDC (no long-lived credentials)
- Separate dev/prod environments
- Production approval workflow
- Minimal IAM permissions

### ⚠️ What You Should Do

- Enable branch protection on `main`
- Require pull request reviews
- Enable status checks
- Use GitHub Environments for production
- Rotate AWS credentials regularly

## Advanced Configuration

### Deploy to Different Regions

Update `AWS_REGION` secret or add region-specific secrets:
- `AWS_REGION_DEV`: ap-northeast-1
- `AWS_REGION_PROD`: us-east-1

### Deploy to Different AWS Accounts

Create separate OIDC roles in each account and use environment-specific secrets:
- `AWS_ROLE_ARN_DEV`: Role in dev account
- `AWS_ROLE_ARN_PROD`: Role in prod account

### Custom Deployment Conditions

Edit `.github/workflows/deploy.yml` to add conditions:
```yaml
if: github.event_name == 'push' && github.ref == 'refs/heads/main'
```
